파이썬으로 시뮬레이션 엔진을 만들고 있습니다

안녕하세요, 허재영입니다. 파이썬으로 복잡한 의사결정 로직을 다루는 시뮬레이션 시스템을 만들고 있습니다. 한 사람 머릿속에 있는 암묵적인 판단을 다른 엔지니어도 안전하게 고칠 수 있는 구조로 풀어내는 일을 많이 해왔습니다. 화려한 아키텍처보다 망가졌을 때 고칠 수 있는 코드를 좋아합니다. 숫자를 먼저 보고 판단하는 편이고, 확신 없이 붙인 복잡성은 몇 주가 걸렸어도 그냥 되돌립니다.

Python · pandas/NumPy · SQLAlchemy · ONNX · PostgreSQL · Dash/Streamlit · GitHub Actions

  1. 완결보다 확장을 먼저 생각합니다.

    한 번에 딱 맞추려다 더 고생하는 경우를 많이 봤습니다. 1년 뒤에 다른 엔지니어가 건드릴 때 어디를 손대면 되는지 보이는 구조가 결국 더 잘 버팁니다. 비슷한 경우가 세 번은 보여야 추상화를 시작하는 편입니다.

  2. 숨어 있는 연결을 드러내려고 합니다.

    콜백 체인이나 __getattr__ 프록시처럼 흐름을 가리는 패턴을 별로 좋아하지 않습니다. 코드가 어디로 흐르는지 읽히는 게 디버깅에도 확장에도 편합니다. 엔진의 God class를 분해하면서 152개+ 파일의 프록시를 모두 명시적인 컴포넌트 경로로 바꾼 게 대표적인 작업입니다.

  3. 측정 없이 최적화는 안 합니다.

    감으로 느리다고 생각했던 코드가 실제로는 멀쩡한 경우를 너무 많이 봤어서, 프로파일러부터 돌립니다. 창고 적재 로직을 104배 빠르게 만들었을 때도 "여기가 느릴 것 같다"가 아니라 측정 결과에서 출발했습니다.

  4. 되돌릴 수 있는 변경을 좋아합니다.

    1,600개 넘는 커밋에 revert가 11번 있습니다. 많은 편은 아닌데, 몇 주 쓴 작업이라도 효과가 없으면 그냥 되돌립니다. ECS로 가봤다가 조회 경로가 이상해져서 facade로 되돌아온 적도, coordinator를 과하게 쪼갰다가 실익 없어서 원복한 적도 있습니다. 중요한 건 코드가 아니라 결과라고 생각해요.

하드코딩된 규칙 → 플러그인 아키텍처

Before

타입이 하나 추가될 때마다 기존 코드를 통째로 복사해서 고치는 구조였습니다. 규칙 하나 바꾸려면 실행 흐름 전체를 건드려야 했고요.

After

상태·의사결정·실행을 세 계층으로 나눠서, 계층을 조합해 새 타입을 만드는 플러그인 구조로 바꿨습니다. 지금은 10종 넘는 타입이 같은 프레임워크 위에서 돌고 있습니다.

  • 상태 계층은 추상 인터페이스로 공통 계약을 잡고, 타입별 차이는 상속이 아니라 조합으로 표현했습니다. 9개 변형이 운영 중입니다
  • 계획 계층은 의사결정 생명주기를 관리하는 ABC이고, 7개 구체 플래너(~3,230줄)가 상속합니다
  • 실행 계층은 template method로 추상화했고, 타입별 실행기는 런타임에 동적으로 등록됩니다
  • 처음부터 일반화하지 않았습니다. 5개 타입을 먼저 각각 구현하면서 공통 생명주기가 드러난 시점에 중복 ~1,442줄을 일반화 ~1,404줄로 교체했습니다

그리디 할당 → 다중 자원 트랜잭션

Before

"지금 비어 있는 자원에 작업을 집어넣는" 그리디 방식이었습니다. 여러 자원을 원자적으로 잡을 수 없어서 중간에 실패하면 상태가 꼬인 채로 남았습니다.

After

경로 전체를 하나의 트랜잭션으로 묶어서 예약하고, 실패하면 자동으로 롤백되도록 바꿨습니다. 3개월 정도 안정화하면서 14건 넘는 엣지 케이스를 잡았습니다.

  • 조율자 하나에 자원 타입별 예약기 3종을 붙여서, 각 타입의 제약을 독립적으로 처리합니다
  • 경로 탐색기가 상태 전환 비용을 고려해서 가능한 경로를 정렬합니다
  • 산재되어 있던 trial 필드 20개 이상을 `_real_values` / `_trial_values` 딕셔너리 + context manager 패턴으로 정리했습니다
  • 실패 시도를 캐시해서 중복을 막고, 성공률에 따라 호출 빈도를 조절합니다

God Object → 명시적 소유 + 이벤트 기반 아키텍처

Before

3,800줄짜리 클래스에 속성이 114개, 관심사가 6가지 뒤섞여 있었습니다. 상태는 싱글턴·인스턴스·그룹에 흩어져 있었고, __getattr__ 프록시가 소유권을 감췄습니다. 콜백 체인이라 부수 효과를 추적하기도 어려웠고요.

After

226줄짜리 코어 + 19개 컴포넌트 + 25개 facade로 풀어냈습니다. 프록시는 0개, pub/sub 이벤트 버스(29줄)로 실행과 부수 효과를 완전히 분리했습니다.

  • 9단계에 걸쳐 80건+ 커밋으로 분해했습니다. 싱글턴에 흩어진 상태부터 개별 인스턴스로 회수했습니다
  • 114개 속성을 19개 컴포넌트 데이터클래스(590줄)로 나눴고, 152개+ 파일에서 프록시를 전부 명시적 경로로 마이그레이션했습니다
  • ECS를 먼저 시도했는데 조회 경로가 너무 간접적이 돼서 facade 패턴으로 갈아탔습니다. 과하게 쪼갠 coordinator(453줄, 226줄)도 실익이 없어서 롤백했습니다
  • 752줄 삭제 / 163줄 추가. 이후 새 기능을 붙일 때 기존 흐름을 건드리지 않아도 됩니다

느린 핫 패스 + 블랙박스 → 최적화 + 관찰 가능성

Before

핫 패스가 한 번 실행에 22M~500M번 호출되면서 느렸습니다. 그런데 왜 이런 의사결정이 나왔는지 역추적할 수단이 없었습니다.

After

프로파일링 기반으로 핫 패스를 최적화했고, 의사결정을 원인까지 되짚어 볼 수 있는 도구를 만들었습니다. 최적화보다 관찰 도구를 먼저 붙인 다음 숫자를 보고 판단했습니다.

  • Enum 캐싱(22M+ 호출)·딕셔너리 인덱스·로컬 캐시·커스텀 `__deepcopy__`로 핫 패스를 줄였습니다
  • Gantt 차트 + 이벤트 타임라인으로 의사결정을 원인까지 역추적할 수 있게 만들었습니다
  • 105개 모듈 구분 enum 기반으로 구조화된 로깅 체계를 구축했습니다
  • 창고 적재 로직은 프로파일링 후 104배 빨라졌습니다. 감이 아니라 측정 결과에서 출발한 결과입니다
