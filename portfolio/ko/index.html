하드코딩된 규칙 → 플러그인 아키텍처

Before

새로운 타입을 추가할 때마다 기존 코드를 복제해서 처음부터 작성. 규칙을 바꾸려면 실행 흐름 전체를 건드려야 함.

After

상태, 의사결정, 실행을 분리한 3계층 플러그인 아키텍처. 계층을 조합해 새 타입을 추가하는 구조. 10종 이상이 하나의 프레임워크 위에서 동작.

  • 상태 계층 — 추상 인터페이스가 공통 계약을 정의, 타입별 차이는 조합으로 표현. 9개 변형 운영 중
  • 계획 계층 — 의사결정 생명주기를 관리하는 ABC. 7개 플래너(~3,230줄)가 상속
  • 실행 계층 — template method로 실행 추상화. 타입별 실행기가 런타임에 동적 등록

그리디 할당 → 다중 자원 트랜잭션

Before

"지금 비어 있는 자원에 작업을 넣는" 그리디 방식. 원자적 할당 불가, 실패 시 롤백 불가.

After

경로 전체를 하나의 트랜잭션으로 묶어 원자적으로 예약. 실패 시 자동 롤백.

  • 조율자 + 3종 타입별 예약기로 자원별 제약을 독립 처리
  • 경로 탐색기가 상태 전환 비용을 고려해 최적 경로 정렬
  • trial/rollback 메커니즘으로 실패 시 자동 롤백 + 중복 방지 캐시

God Object → 명시적 상태 소유 + 이벤트 기반 아키텍처

Before

3,800줄, 114개 속성, 6가지 관심사가 뒤섞인 God Class. __getattr__ 프록시가 소유권을 감추고, 콜백 체인으로 부수 효과 추적 불가.

After

226줄 코어 + 19개 컴포넌트 + 25개 facade. 프록시 0개. pub/sub 이벤트 버스(29줄)로 완전 분리.

  • 114개 속성 → 19개 컴포넌트 + 25개 facade. 152개+ 파일에서 프록시 제거
  • pub/sub 이벤트 버스(29줄)로 실행과 부수 효과 완전 분리
  • 752줄 삭제, 163줄 추가. 새 기능 추가 시 기존 흐름 수정 불필요

느린 핫 패스 + 블랙박스 → 최적화 + 관찰 가능성

Before

핫 패스가 1회 실행당 22M~500M회 호출. 의사결정 원인을 추적할 수단 없음.

After

프로파일링 기반 핫 패스 최적화. 의사결정을 원인까지 역추적할 수 있는 분석 도구 구축.

  • Enum 캐싱(22M+회)·딕셔너리 인덱스·로컬 캐시·커스텀 __deepcopy__로 핫 패스 최적화
  • Gantt 차트 + 이벤트 타임라인으로 의사결정을 원인까지 역추적
  • 105개 모듈 구분 enum 기반 구조화된 로깅 체계 구축
