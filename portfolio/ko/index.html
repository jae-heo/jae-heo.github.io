복잡한 의사결정 로직을 확장 가능하고 관찰 가능한 소프트웨어로 만드는 일을 합니다.

Python 기반 시뮬레이션 엔진에서 18개월간 약 1,300건의 커밋을 단독 수행하며, 플러그인 프레임워크 설계, 레거시 아키텍처 분해, 수천만 회 호출되는 핫 패스 최적화, 그리고 이 모든 것을 디버깅할 수 있는 도구를 만들었습니다. 모든 작업의 동기는 하나였습니다 — 다른 엔지니어가 시스템의 전체 복잡도를 이해하지 않아도 쉽게 기능을 확장할 수 있는 환경을 만드는 것. 개인의 노하우를 공통 프레임워크로, 암묵적 관행을 명시적 구조로 바꾸는 일을 해왔습니다.

Python · pandas/NumPy · SQLAlchemy · ONNX · PostgreSQL · Dash/Streamlit · GitHub Actions/ECR

하드코딩된 규칙 → 플러그인 아키텍처

Before

새로운 타입을 추가할 때마다 기존 코드를 복제해서 처음부터 작성. 규칙을 바꾸려면 실행 흐름 전체를 건드려야 함.

After

상태, 의사결정, 실행을 분리한 3계층 플러그인 아키텍처. 계층을 조합해 새 타입을 추가하는 구조. 10종 이상이 하나의 프레임워크 위에서 동작.

엔진은 서로 다른 제약과 동작을 가진 다양한 처리 모델을 지원해야 했습니다. 기존 방식은 타입마다 코드를 복제해서 커스터마이징하는 것이었고, 타입이 늘어나면서 금방 한계에 부딪혔습니다. 세 가지 관심사를 조합 가능한 계층으로 분리하는 프레임워크를 설계했습니다.

  • 상태 계층 — 의사결정에 필요한 데이터를 소유. 추상 인터페이스가 공통 계약을 정의하고, 타입별 차이는 상속이 아닌 조합으로 표현. 9개의 구체 변형이 운영 중.
  • 계획 계층 — 상태 계층을 소유하고 의사결정 생명주기를 관리하는 ABC. 7개의 구체 플래너(총 ~3,230줄)가 상속.
  • 실행 계층 — template method 패턴으로 실행을 추상화. 타입별 실행기가 런타임에 동적으로 등록.

처음부터 일반화하지 않았습니다. 5개 타입에 먼저 구현해 제약을 검증한 뒤, 공통 생명주기가 드러난 시점에 ~1,442줄의 중복을 제거하고 ~1,404줄의 일반화 코드로 교체했습니다. ONNX 추론도 이 프레임워크 안에 통합했습니다.

그리디 할당 → 다중 자원 트랜잭션

Before

"지금 비어 있는 자원에 작업을 넣는" 그리디 방식. 여러 자원에 걸친 원자적 할당이 불가능하고, 중간에 실패하면 변경된 상태를 되돌릴 수 없음.

After

경로 전체를 하나의 트랜잭션으로 묶어 원자적으로 예약하고, 실패 시 자동 롤백.

작업이 여러 자원에 걸쳐 있을 때, 하나씩 잡아가는 방식으로는 중간에 실패하면 시스템이 불일치 상태에 빠집니다. 자원 타입별 예약 책임을 분리한 뒤 하나의 트랜잭션으로 묶는 구조를 설계했습니다.

  • 조율자 — 전체 예약 흐름을 관리하는 최상위 컨트롤러
  • 타입별 예약기 — 3종의 자원 타입에 대해 각각 전용 예약기로 제약을 독립 처리
  • 경로 탐색기 — 상태 전환 비용을 고려해 가능한 경로를 탐색·정렬
  • trial/rollback 메커니즘 — begin_trial()로 시도, 성공하면 apply_trial()로 확정, 실패하면 자동 롤백
  • 실패 시도 중복 방지 캐시, 성공률 기반 적응형 빈도 조절

3개월간 안정화를 거치며 14건 이상의 엣지 케이스를 해결했습니다.

God Object → 명시적 상태 소유 + 이벤트 기반 아키텍처

Before

3,800줄, 114개 속성, 6가지 관심사가 뒤섞인 God Class. 상태가 싱글턴·인스턴스·그룹에 흩어져 있고 __getattr__ 프록시가 소유권을 감춤. 콜백 체인으로 부수 효과 추적 불가.

After

226줄의 코어 클래스 + 19개 컴포넌트 + 25개 facade. 프록시 0개. pub/sub 이벤트 버스(29줄)로 실행과 후속 처리 완전 분리.

80건+ 커밋, 9단계에 걸쳐 God Object를 점진적으로 분해하고, 콜백 체인을 이벤트 기반으로 전환했습니다.

  • 싱글턴에 흩어진 상태를 개별 인스턴스로 회수
  • mixin 추출 → 114개 속성을 19개 컴포넌트 데이터클래스(590줄)로 분리
  • 6개 기반 facade + 25개 타입별 facade로 실행 로직 재구성
  • 152개+ 파일에서 __getattr__ 프록시를 모두 제거, 명시적 컴포넌트 경로로 마이그레이션
  • pub/sub 이벤트 버스(29줄) — 타입화된 이벤트만 발행, processor를 동적으로 붙이거나 떼는 구조
  • 이력 로깅을 전용 observer로 분리해 부수 효과 완전 제거
  • 752줄 삭제, 163줄 추가. 새 기능 추가 시 기존 흐름 수정 불필요

ECS를 먼저 시도했지만 도메인에 비해 조회 경로가 과도하게 간접적이 되어 facade 패턴으로 전환. 과도한 coordinator 분해도 실익 없어 롤백.

느린 핫 패스 + 블랙박스 → 최적화 + 관찰 가능성

Before

핫 패스가 1회 실행당 22M~500M회 호출되면서 느려지고, 의사결정 원인을 추적할 수단이 없음.

After

프로파일링 기반 집중 최적화로 핫 패스 비용 절감. 의사결정을 원인까지 역추적할 수 있는 분석 도구 구축.

전체 재설계 대신 측정 가능한 병목에 집중했습니다.

  • Enum .value 조회를 모듈 수준 상수로 캐싱 (22M+회 오버헤드 제거)
  • DataFrame boolean mask를 사전 구축 딕셔너리 인덱스로 교체
  • 핫 패스 로컬 캐시 (22M+회, 24M+회)
  • 커스텀 __deepcopy__ — shallow copy + 필요한 필드만 deep copy
  • PostgreSQL 네이티브 COPY ... FROM STDIN WITH CSV
  • Gantt 차트 분석기 — Dash/Streamlit 앱으로 재실행 없이 의사결정 시각적 디버깅
  • 105개 모듈 구분 enum 기반 구조화된 로깅
  • 이벤트 기반 타임라인 추적 — 의사결정의 원인을 끝까지 따라갈 수 있는 구조

공유 메모리 비동기 오프로딩(1,920줄)은 커플링 과도로 판단해 제거. GitHub Actions + ECR CI/CD 파이프라인 구축, 레거시 커넥터 1,915줄 제거 등 운영 인프라 정비도 수행.
